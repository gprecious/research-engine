# research-engine vs Claude Code bench — 2026-04-27 findings

## TL;DR

**Baseline beats research-engine by 3.6 points on average** (RE 79.6 / Baseline 83.2 / Δ -3.6) across 5 topics × 2 trials each, judged by Claude Haiku 4.5 on a 5-axis 0-10 rubric. RE wins on `blog` (+10) and `topic` (+4); ties on `github` (0); loses badly on `arxiv` (-16) and `youtube` (-16).

This is a **net negative signal for the published `research-engine@gprecious-marketplace` v0.8.2 plugin** as evaluated against vanilla Claude Code with a general-purpose subagent.

## Improvement opportunities (P1-P3)

### P1 — Citation Quality is research-engine's biggest weakness

Across 4 of 5 topics, judges flagged RE citations as:
- "minimal source attribution"
- "decorative citations not tied to specific claims"
- "claims without attribution"

The arxiv judge gave RE 5/10 on citation while baseline got 9/10 — a 4-point gap on a single topic. The youtube judge similarly: RE 5 vs Baseline 8.

**Action**: enforce per-claim citation in `agents/*-adapter.md` outputs. The adapter prompts currently require `[src]` markers but adapters often ship findings with weak claim → source linkage. Tighten the contract: every body sentence must end in a `[n]` marker or be excluded.

### P1 — arxiv-adapter under-uses paper body content

For the Mamba paper, RE produced 1517 / 1546 word reports (run1 / run2) while baseline produced 2983 / 2752 word reports — baseline was nearly 2× longer with much higher technical density (full 28-numbered-citation bibliography, mathematical formulations, hardware kernel discussion).

Judge rationale: RE "explains concepts at overview level without mathematical rigor or detailed mechanistic analysis" while baseline provides "mathematical formulation, explicit theorem, hardware optimization details, and mechanistic insights grounded in established theory."

The arxiv-adapter may be relying too heavily on the paper's abstract and not pulling enough from the body PDF or HTML version. Investigate: does the adapter actually fetch the PDF, or only abstract page metadata?

**Action**: arxiv-adapter should fetch and synthesize from paper body (sections 3–5 typically), not just abstract + first 2 KB.

### P1 — Limitations section fix from this PR did not reach the runtime

Because the bench invokes `/research` via `Skill('research-engine:research')` which loads the published plugin from `~/.claude/plugins/cache/gprecious-marketplace/research-engine/0.8.2/`, our worktree's modification to `lib/report_sections.md` (commit 81dfb13) was NOT applied during these runs. The youtube judge still flagged RE for "limited critical analysis or limitations discussion" — the exact gap the §7 fix targets.

**Action**: re-run this bench after the next research-engine release (v0.9.0+) to validate the Limitations fix. Expected: youtube delta improves from -16 toward break-even.

### P2 — Depth is the second-largest axis gap (-1.2 baseline-favoring)

Depth lagged behind baseline on arxiv and youtube. The pattern: RE stays at "intermediate / intuition-building" level while baseline pushes into mechanism specifics, formal definitions, ablation findings, and explicit limitation analysis.

**Action**: audit the synthesis stage prompt in `commands/research.md:Stage 5` to push for "what would surprise a domain expert?" rather than "what would a curious newcomer want to know?" The current prompt may be over-indexed on accessibility.

### P3 — RE's strengths are real but topic-dependent

RE's clear wins:
- **Blog post analysis** (+10 on Anthropic harness): RE's quotation-anchor pattern + bilingual TL;DR + frontmatter shines on well-structured prose. Judge: "full quotation anchor [1] for every substantive point... exact cost tables ($124.70 itemized)... concrete Evaluator failure catches".
- **Structure axis** (+0.2): consistent positive across all 5 topics. RE's frontmatter + standard sections give it a small but reliable structural lead.
- **Reproducibility** (+0.6): RE's templates produce more consistent run-to-run outputs than baseline's free-form synthesis. Notable since the spec hypothesis was that RE might be LESS reproducible due to adapter dispatch nondeterminism — opposite was observed.

**Action**: lean into these strengths. Consider whether the RE pipeline can be triggered for blog-style inputs more aggressively (it already is — the blog adapter handles this category).

### P3 — `topic-moe-routing-2025` reproducibility low (RE 6, Baseline 5)

Open-ended topic queries had the lowest reproducibility scores in the matrix (6 and 5 vs 7-9 for URL-anchored topics). This is expected — without a fixed input, web search results and adapter dispatch decisions vary more between runs.

**Action**: not a defect but worth documenting in `/research`'s usage guidance — "topic mode" intrinsically less reproducible than URL mode.

## Bench harness notes (separate from research-engine improvements)

During this matrix run we discovered and fixed:
- `collect_metrics.sh` aborted under `set -euo pipefail` when `grep -oE '\[[0-9]+\]'` had zero matches — fixed in commit `b0c96f6`.
- Three RE/run2 subagents skipped the bench-output copy step after /research completed; we recovered by manual `cp` + `collect_metrics`. Suggests we should make the per-run subagent prompt template more emphatic about post-/research bookkeeping (or simpler: wrap it in a single shell snippet at the end of the prompt that the subagent just runs).
- `/research`'s Notion adapter trips over commas in select fields multiple times across runs (3 of 10 RE runs reported this). Existing /research bug — not bench-specific. Consider sanitizing `intent` strings before pushing to Notion.

## Methodology

- **Topics**: 5 (1 per adapter category).
- **Trials**: N=2 per (topic, mode).
- **Judge**: Claude Haiku 4.5 dispatched via Agent tool (in-session orchestration; external `claude -p` was rate-limited so judge.py's call_claude path was bypassed for this run).
- **Blinding**: A/B labels randomized per dispatch; no judge response leaked label-position keywords.
- **Metrics**: per-run `meta.json` (word_count, citation_count, external_link_count). `model_tokens` recorded as null because `--output-format=json` capture is not used in in-session orchestration.
- **Aggregation**: weighted total = average of 5 axes × 10. Per-topic delta = RE − baseline.

## Limitations of this bench

(re-stated from the auto-generated report)
- LLM-as-judge with same model family (Claude): potential self-favoring bias.
- N=2 trials per (topic, mode): too small for statistical confidence intervals.
- 5 topics = 1 per category: weak generalization within categories.
- `baseline_prompt` phrasing is sensitive — small wording changes can move scores.
- Plugin version tested is **the published 0.8.2** loaded from `~/.claude/plugins/cache/`, not the worktree HEAD. Improvements committed on this branch (e.g., the Limitations section fix) take effect only after the next plugin release.
